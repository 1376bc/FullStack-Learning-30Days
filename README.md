## 📝 技术心得汇总

### 1. 工业级后端架构 (Day 18 重构重点)
- **三层架构 (3-Tier)**: 实现了 Controller (接客) -> Service (加工) -> Repository (仓库) 的解耦，提升了代码的复用性和可维护性。
- **自动化文档 (Swagger)**: 集成 SpringDoc OpenAPI，实现了接口文档的自动生成与在线调试，彻底告别手动编写接口文档。
- **全局异常拦截**: 使用 `@RestControllerAdvice` 构建了系统的“防弹衣”，确保无论后端出什么错，前端都能收到格式统一的友好提示。

### 2. 多媒体文件处理 (Day 16)
- **动静分离存储**: 掌握了 `MultipartFile` 处理文件流。图片本体存放在硬盘（物理路径），数据库仅存相对路径（虚拟路径）。
- **静态资源映射**: 通过 `WebMvcConfigurer` 将浏览器的 URL 映射到服务器真实的图片文件夹。

### 3. 数据分析与异构系统 (Day 15)
- **异构架构**: 实现了 Java 处理实时业务，Python 处理离线报表。通过 FastAPI 搭建微服务，打通了 `MySQL -> Pandas -> Excel` 的链路。

## ⚠️ 避坑指南 (开发者血泪史)

### 🔴 Spring Boot 3 与 Swagger 专题
1. **Swagger 404 陷阱**: 手动配置静态资源路径时，极易覆盖 Jar 包内部路径。
   - **解决**: 建议在 Java 配置类中显式指定 `classpath:/META-INF/resources/`。
2. **拼写致命伤**: `META-INF` 误写为 `META-IF` 会导致所有 Swagger 资源加载失败。
3. **注解语法**: Java 注解（如 `@ExceptionHandler`）末尾千万**不能加分号**，否则会导致方法解析失败。

### 🟡 Git 与 环境
1. **Git 冲突解决**: 遇到 `rejected` 需先 `pull`。若弹出黑色 Vim 界面，按 `Esc` 后输入 `:wq` 退出。
2. **孤儿注解**: `@Autowired` 下方若变量被注释，注解会错误作用于下一个方法，导致“找不到 Bean”报错。
3. **代理冲突**: 开启代理时，需配置 Git 走特定端口；若关闭代理，务必执行 `unset http.proxy` 否则会连接超时。

### 🔵 前端细节
1. **JS 引号陷阱**: `${}` 变量嵌入必须使用 **反引号 (`` ` ``)**。
2. **FormData 传输**: 包含图片上传的请求必须使用 `FormData` 对象，且不能手动设置 `Content-Type`（由浏览器自动设置 boundary）。

## 👨‍💻 面试官提问模拟 (Interview Q&A)

**Q1：为什么在项目里引入 Service 层？直接在 Controller 写 SQL 不快吗？**
> **A**：在小练习里确实快，但在工程中，Service 层有两个核心价值：
> 1. **事务管理**：`@Transactional` 只能在 Service 层生效，保证多步数据库操作的原子性。
> 2. **逻辑复用**：同一个业务逻辑（如计算学生总分）可以被网页端、APP端、定时任务同时调用，避免代码重复。

**Q2：如何处理用户上传超大图片导致的服务器压力？**
> **A**：我在 `application.properties` 中设置了 `max-file-size: 5MB` 进行初步拦截。在更专业的生产环境中，我会使用 **Nginx** 在流量进入 Java 前直接拦截，或者将图片直接上传到 **OSS (对象存储)**，减轻应用服务器的带宽压力。

**Q3：你的全局异常处理器是如何工作的？**
> **A**：它像一个“切面”。通过 `@ExceptionHandler` 捕获特定的异常（如 `RuntimeException` 或文件过大异常），然后将复杂的错误堆栈包装成一个统一的 JSON 格式（包含 success, message 等字段）返回。这不仅保护了后端代码结构，也极大地优化了前端的报错提示。
